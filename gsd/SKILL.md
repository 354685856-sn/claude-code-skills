# GSD (Get Stuff Done) - 高效执行框架

## 核心理念

将复杂任务拆分为三个明确阶段，避免上下文丢失和执行偏差。

---

## 三阶段流程

### 阶段一：PLAN（计划）

**目标**：明确需求、设计方案、拆解任务

```
1. 需求澄清
   - 用户想要什么？
   - 成功标准是什么？
   - 有哪些约束条件？

2. 架构设计
   - 涉及哪些文件/模块？
   - 数据流和控制流如何？
   - 需要哪些外部依赖？

3. 任务拆解
   - 拆分为可独立验证的小任务
   - 估算每个任务的工作量
   - 确定执行顺序
```

**输出**：`gsd_plan.md` 文件，包含：
- 项目概述
- 技术方案
- 任务清单（带复选框）
- 验收标准

---

### 阶段二：EXECUTE（执行）

**目标**：按任务清单逐项完成

```
1. 逐个完成任务清单中的任务
2. 每完成一个任务立即标记
3. 遇到问题及时记录并寻求反馈
4. 保持代码与设计方案一致
```

**输出**：`gsd_execution_log.md` 文件，包含：
- 已完成任务列表
- 遇到的问题及解决方案
- 设计变更记录

---

### 阶段三：REVIEW（审查）

**目标**：验证完成质量、确保无遗漏

```
1. 对照验收标准逐项检查
2. 运行测试验证功能
3. 代码审查（命名、结构、注释）
4. 确认所有任务完成
```

**输出**：`gsd_review.md` 文件，包含：
- 验收结果（PASS/FAIL）
- 遗留问题列表
- 后续改进建议

---

## 使用方式

### 启动新项目
```
/gsd 新功能需求描述
```

### 继续执行
```
/gsd continue
```

### 检查状态
```
/gsd status
```

### 完成审查
```
/gsd review
```

```
└── active_tasks.json     # 当前任务状态
```

---

## 核心原则

1. **先计划后执行** - 禁止未经设计的编码
2. **任务原子化** - 每个任务应在 30 分钟内完成
3. **即时标记** - 完成即标记，保持状态同步
4. **审查闭环** - 无审查不交付
